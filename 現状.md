# AllCampus テスト計画書

## 1. プロジェクト概要
本ドキュメントは、教育機関向け業務システム「AllCampus」大規模改修案件におけるテスト計画をまとめたものである。

### 1.1 対象システム
*   **システム名**: AllCampus
*   **サブシステム**:
    *   **gakumu**: 教員向け業務システム（約500ページ）
    *   **campus**: 学生向け業務システム（約100ページ）

### 1.2 技術スタック
*   **Frontend**: Vue.js, TypeScript
*   **Backend**: Java, C#, GraphQL

## 2. 現状と課題
### 2.1 スケジュール状況
*   当初納期: 11月
*   現在予定: 3月納品（延期済み）
*   リスク: 現状ペースではさらに約半年の遅延見込みあり

### 2.2 開発プロセスの課題
*   **仕様の不明確さ**: 設計書不在、仕様書が曖昧。業務フローが不明確。
*   **手戻りの多発**: 開発側が仕様解釈に時間を要している。
*   **テスト基準の欠如**: 「完成」の定義が難しく、テストケース作成が困難。

## 3. テスト戦略
### 3.1 テスト方針
*   **UAT（User Acceptance Test）中心**: ユーザー視点での動作確認を重視する。
*   **先行テストの実施**: 複数画面で共通利用される機能については、画面実装完了を待たずに先行してテストを実施する。
*   **テスト基準**:
    *   旧システムの挙動
    *   現存する仕様書（曖昧な部分は開発・PMと協議）
    *   デザイナー作成のUIデザイン（デザイナーリソースに余裕があるため活用推奨）

### 3.2 テストプロセス
1.  **開発完了**: 開発者より「1ページ完成」の連絡。
2.  **テスト実施**: テスターが旧システム・仕様書・デザインをもとに確認。
3.  **フィードバック**: 1画面あたり平均3件程度の指摘を想定。
    **課題**: 確認者による粒度のバラつき -> 指摘基準のすり合わせが必要。

## 4. 今後の対策案
*   **共通機能の早期洗い出し**: 先行テスト対象を明確化し、未開発の機能を洗い出す。
*   **機能の受け入れ基準明確化**: 仕様書不足を補うため、機能に求められる目的をQAサイドで定義する。
*   **フィードバック粒度の統一**: テスター間での認識合わせ、チェックリストの整備。
*   **機能ごとにテストを実施する**: 業務フローが不明であるため、機能ごとにシナリオを定義し、機能ごとのテストを重視する。 
*   **Antigravityによる補助導入**: テストケース生成にはCLIでルール表から引用できるようにし、テスト品質の均一化を図る。
*   **開発側との情報連携**: これまではAsanaが主体で連携が容易であったが、リポジトリで管理する形にするため連携手段を構築する必要がある。
*   **検証環境の担当者**: データ比較には検証環境を使用するため、一人で実行するのか分担するのかを協議する必要がある。
*   **QAによる修正手順確立**: 修正するタスクと、開発者に修正依頼するタスクを明確にし、QAサイドによる修正着手のハードルを下げる。
