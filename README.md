# AllCampus テスト計画

現状では設計書が存在せず、仕様が曖昧であるため、開発時に迷いと手戻りが多くなっている状況で、開発に遅れが生じている現状。
また、業務フローも共有されていないため、シナリオテストが実施できないため、画面ごとではなく機能ごとにシナリオを定義する。
シナリオが定義された機能それぞれのテストを単体テストと捉える。
シナリオが定義された機能がちりばめられたものを画面とし、一画面のテストを結合テストと捉える。

曖昧な仕様と設計を受け入れ基準として定義するため、機能の目的に注視し、目的を果たすことを受け入れ基準とする。
ここでいう目的とは、想定されるユーザーストーリーとUATのフィードバックであり、UI/UXは考慮しないものとする。

## シフトレフトアプローチ
時すでに遅しである。残念。

## テスト分析
* 共通機能か、固有機能かで選別（共通機能の場合は、`共通機能コピペ用.md`から選択し、対応するテストを実施する）
* リスクとリスクレベルを分析し、優先順位を決定
上記の2点をベースをテスト設計書を作成し、成果物としてリポジトリに集約する

## テスト設計
経験ベースのテストを避け、ブラックボックステストの精度を均一化する。

開発前であっても共通機能のみテストを実施し、その時点でそのテストが実行できない場合は中止理由をテスト設計書に記載する。
完了している場合は開発者に共有する目的でテスト設計書に完了と明記する。

TDDは現実的でないため、振る舞い駆動開発（BDD）を採用する。
Given（前提条件）When（実行する行動）Then（期待される結果）の3点を基にテスト仕様書を作成する。
本プロダクトにおいてはThenを機能の目的と捉え、目的を果たしていることを受け入れ基準とする。(目的を果たしていれば、設計を満たしていると判断する)
これは、定義されていない設計を補う目的であり、設計の定義とするものではない。

システムの望ましい振る舞いを、現行版と仕様書を参照し、共通機能のみ[共通機能振る舞いルール表.md]に記載する。(テストチームが担当)
また、UATでクライアント側のフィードバックを参考に、共通機能振る舞いルール表を更新する。


## テスト結果フィードバック
不具合があった場合、共通機能の判定HとLについては開発課題に飛ばすか、QA側で修正を行う。
それ以外の優先度の高い不具合は即時修正するものとし、QA側で修正を行うか、修正が困難な場合は開発者に修正依頼する。

未記載の共通機能については共通機能コピペ用.mdを更新する。
また、コード例として正解となる共通機能のコードを追加する。
